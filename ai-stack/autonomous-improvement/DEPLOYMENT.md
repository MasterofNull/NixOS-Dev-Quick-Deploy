## Deployment Steps

### Step 1: Verify Current State

```bash
# Check that database tables exist
PGPASSWORD=$(cat /run/secrets/postgres_password | tr -d '\n') \
  psql -h 127.0.0.1 -U aidb -d aidb \
  -c "SELECT tablename FROM pg_tables WHERE schemaname = 'public' AND tablename LIKE '%improvement%'"

# Expected output:
#  improvement_cycles
#  improvement_recommendations
#  trigger_events
#  (and more)
```

### Step 2: Rebuild NixOS

This will activate the autonomous improvement service:

```bash
cd ~/Documents/NixOS-Dev-Quick-Deploy
sudo nixos-rebuild switch --flake .#hyperd
```

What happens during rebuild:
1. Nix module creates Python environment with dependencies (psycopg2, aiohttp)
2. Systemd service `ai-autonomous-improvement.service` is created
3. Systemd timer `ai-autonomous-improvement.timer` is created and started
4. CLI wrapper `aq-autonomous-improve` is installed to PATH
5. Service starts checking for anomalies every 60 minutes

### Step 3: Verify Service Activation

After rebuild completes:

```bash
# Check timer is active
systemctl status ai-autonomous-improvement.timer

# Expected output:
#   Active: active (waiting)
#   Next run: <timestamp>

# Check service was created
systemctl cat ai-autonomous-improvement.service | head -20

# Test CLI wrapper
aq-autonomous-improve --help
```

### Step 4: Run Initial Dry-Run Test

Test the complete cycle without making changes:

```bash
# Run one cycle in dry-run mode
aq-autonomous-improve run --dry-run

# Watch the output for:
# 📊 Phase 1: Syncing metrics from all sources...
# 🎯 Phase 2: Checking trigger conditions...
# 🔬 Phase 3: Research - Local LLM generating hypotheses... (if triggered)
```

Expected flow:
1. Syncs metrics from routing_metrics.db, experiments.sqlite
2. Computes trends and detects anomalies
3. If anomalies found → LLM analyzes and decides whether to trigger
4. If triggered → LLM generates optimization hypotheses
5. Dry-run skips actual execution, shows what would be done

### Step 5: Monitor Operations

```bash
# View real-time logs
journalctl -u ai-autonomous-improvement.service -f

# Check recent improvement cycles
aq-autonomous-improve status

# List hypotheses generated by LLM
aq-autonomous-improve hypotheses

# Show metric trends
aq-autonomous-improve trends

# Query PostgreSQL directly
PGPASSWORD=$(cat /run/secrets/postgres_password | tr -d '\n') \
  psql -h 127.0.0.1 -U aidb -d aidb \
  -c "SELECT * FROM improvement_cycles ORDER BY started_at DESC LIMIT 5"
```

### Step 6: Enable Production Mode (After Testing)

Once dry-run testing is successful, enable actual execution:

Edit `nix/hosts/hyperd/facts.nix`:

```nix
autonomousImprovement = {
  enable = true;
  interval = 60;
  dryRun = false;  # ← Change to false for production
};
```

Then rebuild:

```bash
sudo nixos-rebuild switch --flake .#hyperd
```

## Troubleshooting

### Timer Not Running

```bash
# Check timer status
systemctl status ai-autonomous-improvement.timer

# If inactive, start it
sudo systemctl start ai-autonomous-improvement.timer

# Enable for boot
sudo systemctl enable ai-autonomous-improvement.timer
```

### Service Fails to Start

```bash
# View full logs with errors
journalctl -u ai-autonomous-improvement.service -n 50 --no-pager

# Common issues:
# 1. PostgreSQL not running → sudo systemctl start postgresql.service
# 2. llama.cpp not running → sudo systemctl start llama-cpp.service
# 3. Missing secrets → check /run/secrets/postgres_password exists
```

### Python Import Errors

```bash
# Verify Python environment was created correctly
ls -la /nix/store/*python3*-env*/lib/python3.*/site-packages/ | grep -E "(psycopg2|aiohttp)"

# If missing, rebuild should create the environment
sudo nixos-rebuild switch --flake .#hyperd
```

### Database Connection Issues

```bash
# Test PostgreSQL connection manually
PGPASSWORD=$(cat /run/secrets/postgres_password | tr -d '\n') \
  psql -h 127.0.0.1 -U aidb -d aidb -c "SELECT 1"

# Check postgres service is running
systemctl status postgresql.service

# Check port is listening
ss -tlnp | grep 5432
```

### LLM Not Responding

```bash
# Check llama.cpp is running
systemctl status llama-cpp.service

# Test LLM endpoint
curl -s http://localhost:8080/health

# View llama.cpp logs
journalctl -u llama-cpp.service -n 20
```

## Configuration Options

### Interval Adjustment

Change check frequency by editing `interval` in facts.nix:

```nix
autonomousImprovement = {
  enable = true;
  interval = 30;  # Check every 30 minutes instead of 60
  dryRun = false;
};
```

### Manual Trigger

Run a cycle manually at any time:

```bash
# Run one cycle now
aq-autonomous-improve run

# Run in dry-run mode
aq-autonomous-improve run --dry-run
```

### Daemon Mode

Run continuously with custom interval:

```bash
# Run as daemon checking every 30 minutes
aq-autonomous-improve daemon --interval 30
```

### Disable Service

Temporarily disable without removing:

```bash
# Stop timer
sudo systemctl stop ai-autonomous-improvement.timer

# Disable from boot
sudo systemctl disable ai-autonomous-improvement.timer
```

Or set `enable = false` in facts.nix and rebuild.

## What Happens During a Cycle

### 1. Metric Sync (Phase 1)
```
📊 Collecting from routing_metrics.db (cache hits, local routing %)
📊 Collecting from experiments.sqlite (autoresearch success rate)
📊 Computing trends over 1h, 24h, 7d windows
📊 Detecting anomalies (degrading trends, volatility)
```

### 2. Trigger Check (Phase 2)
```
🎯 LLM analyzes anomalies
🎯 Decides: "Should we investigate?"
🎯 If yes → Creates TriggerEvent
🎯 If no → "System healthy, no action needed"
```

### 3. Research (Phase 3 - if triggered)
```
🔬 LLM acts as Research Director
🔬 Analyzes patterns in metrics
🔬 Generates 3-5 optimization hypotheses
🔬 Each with: evidence, experiment config, impact estimate
🔬 Ranks by priority score
```

### 4. Execution (Phase 4 - future)
```
🧪 Currently placeholder
🧪 Will integrate with autoresearch framework
🧪 Run A/B tests: baseline vs variant
🧪 Statistical validation
🧪 Auto-accept if improvement >5%, p<0.05
```

### 5. Learning (Phase 5)
```
✅ Store cycle results to PostgreSQL
✅ Record decisions and reasoning
✅ Update knowledge base
✅ Feed future cycles
```

## Database Tables

The system uses these PostgreSQL tables:

| Table | Purpose |
|-------|---------|
| improvement_cycles | Complete cycle records |
| trigger_events | What triggered each cycle |
| optimization_hypotheses | LLM-generated ideas |
| experiment_results_history | A/B test outcomes |
| system_metrics_timeseries | Aggregated metrics |
| metric_trends | Computed trends (1h/24h/7d) |
| design_decisions | Architectural choices |
| improvement_recommendations | Pending suggestions |
