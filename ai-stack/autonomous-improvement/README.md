# Autonomous Improvement System

**Phase 1: Local LLM-Driven Autonomous Optimization** (COMPLETE)

This system transforms the local LLM from a passive responder into an **active decision-maker** that autonomously improves system performance.

## Components

### 1. Trend Database (`trend_database.py`)
- Aggregates metrics from multiple sources:
  - `routing_metrics.db` (cache hit rate, local routing %)
  - `experiments.sqlite` (autoresearch results)
  - Baseline profiler (future integration)
- Computes trends over time windows (1h, 24h, 7d)
- Detects anomalies (degrading trends, high volatility)
- Stores to PostgreSQL for LLM analysis

**Key Functions:**
```python
async def sync_metrics_pipeline(since_hours: int = 24) -> Dict[str, int]:
    """Collect → Insert → Analyze → Cache trends"""

def detect_anomalies() -> List[Dict[str, Any]]:
    """Find metrics that warrant improvement cycles"""
```

### 2. Trigger Engine (`trigger_engine.py`)
- Uses local LLM to analyze anomalies
- LLM decides: "Should we trigger an improvement cycle?"
- Records all decisions with reasoning
- Cooldown mechanism prevents duplicate triggers

**LLM Decision Flow:**
```python
anomalies = trend_db.detect_anomalies()
→ LLM analyzes context
→ Returns: {should_trigger, reasoning, severity, focus_area}
→ If true: Creates TriggerEvent, launches improvement cycle
```

### 3. Research Phase (`research_phase.py`)
- Local LLM acts as "Research Director"
- Analyzes patterns and generates optimization hypotheses
- Each hypothesis includes:
  - Type (CONFIG_TUNING, PROMPT_OPTIMIZATION, etc.)
  - Evidence from metrics
  - Experiment config (baseline vs variant)
  - Estimated impact
  - Risk level and priority score

**Hypothesis Generation:**
```python
hypotheses = await research_phase.conduct_research(trigger_id)
→ Gathers context (anomalies + trends)
→ LLM generates 3-5 testable hypotheses
→ Ranks by priority
→ Stores to PostgreSQL
```

### 4. Autonomous Loop (`autonomous_loop.py`)
- Main orchestrator tying all components together
- Implements complete improvement cycle:
  1. Sync metrics from all sources
  2. Check for trigger conditions
  3. Research (if triggered)
  4. Execute experiments (future: integrate with autoresearch)
  5. Validate and learn

**Cycle Execution:**
```python
cycle = await autonomous_loop.run_improvement_cycle()
→ Sync metrics (trend_database)
→ Check triggers (trigger_engine)
→ Generate hypotheses (research_phase)
→ Execute experiments (placeholder)
→ Validate and apply changes
→ Record to PostgreSQL
```

### 5. CLI Wrapper (`aq-autonomous-improve`)
User-friendly commands for operating the system:

```bash
# Run one improvement cycle
aq-autonomous-improve run

# Run as daemon (continuous mode)
aq-autonomous-improve daemon --interval 30

# Dry run (research only, no execution)
aq-autonomous-improve run --dry-run

# Show recent cycles
aq-autonomous-improve status
aq-autonomous-improve status --json

# List hypotheses generated by LLM
aq-autonomous-improve hypotheses

# Show metric trends
aq-autonomous-improve trends
```

5. **system_metrics_timeseries** - Aggregated metrics
6. **metric_trends** - Computed trends over time windows

## Deployment

### Step 1: Enable in NixOS Configuration

Add to your NixOS configuration:

```nix
{
  mySystem.aiStack.autonomousImprovement = {
    enable = true;
    interval = 60;  # Check every 60 minutes
    dryRun = false; # Set to true for research-only mode
  };
}
```

### Step 2: Rebuild NixOS

```bash
sudo nixos-rebuild switch
```

This will:
- Create systemd service `ai-autonomous-improvement.service`
- Create systemd timer `ai-autonomous-improvement.timer`
- Install CLI wrapper `aq-autonomous-improve`
- Set up Python environment with dependencies (psycopg2, aiohttp)

### Step 3: Verify Service

```bash
# Check timer status
systemctl status ai-autonomous-improvement.timer

# Check service status
systemctl status ai-autonomous-improvement.service

# View logs
journalctl -u ai-autonomous-improvement.service -f

# Run manual cycle
aq-autonomous-improve run --dry-run
```

### Step 4: Monitor Operations

```bash
# View recent improvement cycles
aq-autonomous-improve status

# List hypotheses generated by LLM
aq-autonomous-improve hypotheses

# Show metric trends
aq-autonomous-improve trends

# Check PostgreSQL directly
psql -U aidb -d aidb -c "SELECT * FROM improvement_cycles ORDER BY started_at DESC LIMIT 5"
```

## How It Works

### Example: Cache Hit Rate Degradation

1. **Detection** (every 60 minutes)
   ```
   Metric: cache_hit_rate
   Baseline: 0.85 (24h ago)
   Current: 0.70 (now)
   Change: -17.6% → ANOMALY
   ```

2. **LLM Analysis**
   ```
   Trigger Engine: "Cache hit rate degraded 17.6%.
                    This warrants investigation."
   Decision: TRIGGER improvement cycle
   Severity: high
   ```

3. **Research Phase**
   ```
   LLM Research Director generates hypotheses:

   1. [CONFIG_TUNING] Priority: 0.85
      "Cache TTL too short causing frequent evictions"
      Evidence: 15% drop every 5min window
      Experiment: Increase TTL from 300s to 900s
      Expected: 20% reduction in cache misses
      Risk: low

   2. [CACHING_STRATEGY] Priority: 0.75
      "Semantic similarity threshold too strict"
      Evidence: High query diversity, low hit rate
      Experiment: Lower threshold from 0.95 to 0.90
      Expected: 15% improvement in hit rate
      Risk: medium
   ```

4. **Execution** (future integration with autoresearch)
   ```
   Run A/B test:
   - Baseline: current config
   - Variant: TTL=900s

   Measure cache_hit_rate over 1 hour
   Statistical validation: p<0.05

   If improvement >5%:
     → Auto-accept
     → Apply to production
     → Record to PostgreSQL
   ```

5. **Learning**
   ```
   Store to PostgreSQL:
   - Hypothesis ID
   - Experiment results
   - Statistical significance
   - Applied changes
   - Observed improvement

   Use for future decisions
   ```

## Security

- **Secrets**: PostgreSQL password via SOPS (`/run/secrets/postgres_password`)
- **Sandboxing**: Systemd hardening (ProtectHome, ReadOnlyPaths, etc.)
- **Network**: Localhost-only (127.0.0.1)
- **User**: Runs as primary user, not root
- **Audit**: All decisions recorded to PostgreSQL
