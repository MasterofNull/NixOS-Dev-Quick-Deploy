# AI Harness Routing and Editor Surfaces
Status: Active
Owner: AI Stack Maintainers
Last Updated: 2026-05-07

Purpose:
- document the current local, remote, and harness routing model
- make the VSCodium wrapper and extension surfaces explicit
- clarify which token limits apply to user prompts versus agent-to-agent traffic
- give operators one command-center-oriented map for the active AI path

## Routing Model

There are three distinct routing layers. They serve different jobs and should
not be treated as interchangeable.

### 1. Local interactive lane

Primary surfaces:
- Continue IDE chat in VSCodium
- local editor autocomplete
- compact local assist flows

Authoritative ingress:
- switchboard OpenAI-compatible endpoint on `http://127.0.0.1:8085/v1`

Primary profiles:
- `continue-local`
- `embedded-assist`

Intent:
- keep user-facing editor interactions fast, private, and bounded
- prefer local inference first
- avoid spending remote tokens for simple IDE work

### 2. Remote interactive lane

Primary surfaces:
- Continue model picker in VSCodium
- remote coding and reasoning lanes through switchboard

Authoritative ingress:
- same switchboard OpenAI-compatible endpoint on `http://127.0.0.1:8085/v1`

Primary profiles:
- `remote-coding`
- `remote-reasoning`
- `remote-free`
- `remote-gemini`

Intent:
- keep remote provider usage behind switchboard policy
- avoid putting provider secrets into editor configs
- route remote work by task class rather than by ad hoc client setup

### 3. Harness workflow lane

Primary surfaces:
- `aq-*` CLI tools
- hybrid-coordinator workflow endpoints
- persisted workflow runs, reviewer gates, and delegated sub-agent sessions

Authoritative ingress:
- hybrid-coordinator on `http://127.0.0.1:8003`

Primary APIs:
- `/workflow/plan`
- `/workflow/run/start`
- `/hints`
- `/review/acceptance`
- `/health`

Intent:
- own orchestration, review, persisted run state, and evidence
- carry the agent-to-agent traffic that needs aggressive context control
- remain the primary interface for planning and multi-step execution

## Budget Semantics

This distinction matters:

- User-facing editor prompts are not governed by the same compact default output
  caps that the harness uses for internal agent traffic.
- Switchboard profile `maxOutputTokens` values are profile defaults and routing
  policy hints. They are not universal hard caps for every interactive client.
- Agent-to-agent budgets are controlled by workflow/session policy, especially
  `token_limit`, tool-call limits, and blueprint/session orchestration rules.

Current intent:
- Continue interactive chat gets a larger editor-oriented budget.
- local compact agent lanes remain tighter.
- workflow/session limits remain the authoritative control for internal
  delegation noise, reviewer loops, and context explosion prevention.

## VSCodium Surfaces

Declarative source of truth:
- `nix/home/base.nix`

Managed pieces:
- VSCodium wrapper PATH and environment
- installed AI extensions
- writable `~/.config/VSCodium/User/settings.json`
- generated `~/.continue/config.json`

Key editor surfaces:
- Continue IDE extension
- Codex extension and CLI executable wiring
- Claude extension executable wiring
- Gemini companion and Code Assist extensions
- Qwen IDE companion extension

Operational rule:
- editor clients should point at switchboard and harness-owned surfaces, not at
  raw remote providers or hardcoded secrets

## Continue Model Map

Generated by Home Manager:
- `Local Chat (continue-local)`
- `Local Agent (Harness-Aware)`
- `Remote Coding (Switchboard)`
- `Remote Reasoning (Switchboard)`
- `Remote Free (Switchboard)`
- `Remote Gemini (Switchboard)`

Meaning:
- local user chat stays on `continue-local`
- harness-heavy local work should move to `local-agent`
- remote lanes stay behind `X-AI-Profile` headers instead of direct provider
  credentials in the editor

## Rebuild and Switch Note

If `nixos-rebuild switch --flake .#hyperd-ai-dev` fails with:

```text
creating symlink ... /nix/var/nix/profiles/system-... Permission denied
```

that indicates an activation privilege problem while updating the system
profile. It is not a flake evaluation failure.

Use:

```bash
nix build .#nixosConfigurations.hyperd-ai-dev.config.system.build.toplevel --no-link
sudo nixos-rebuild switch --flake .#hyperd-ai-dev
home-manager switch --flake .#hyperd
```
