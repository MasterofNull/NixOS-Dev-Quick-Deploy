## Architecture Decisions (from review)

### A1 — MCP Tool Priority (Gemini, Codex, Qwen3 consensus)
The MCP tool (`context_system_state`) is the highest-value slice. Current sessions spend
~500 tokens on re-deriving service/error state. The tool returns this in one call.
Dashboard is deferred — agents benefit immediately; humans can wait one cycle.

**Revised priority: 115.4 MCP tool → 115.5 session-start → 115.3 dashboard**

### A2 — No Synchronous Refresh (Gemini, Codex, Qwen3 consensus)
The MCP tool MUST NOT synchronously spawn `aq-system-state` when the artifact is stale.
Doing so blocks the coordinator aiohttp event loop (CRITICAL).

**Decision**: Return the last valid artifact with a `_stale: true` marker. Trigger an
async background refresh via the systemd timer path (POST to a refresh endpoint or
subprocess with `asyncio.to_thread`). Never block the response on generation.

### A3 — Concurrency Lockfile (all agents)
Multiple concurrent agents calling `context_system_state` or triggering refresh would
cause thundering herd. A single lock prevents simultaneous artifact generation.

**Decision**: Use `/var/lib/ai-stack/hybrid/telemetry/.system-state.lock` as a
`fcntl` lockfile. Readers never take the lock; only the writer (aq-system-state) takes it.
MCP tool reads the artifact without locking — atomic writes guarantee consistency.

### A4 — Progressive Disclosure (Gemini, Codex)
The full artifact JSON is large (50-200KB). The MCP tool MUST default to a compressed
≤200-token text summary. Domain-filtered JSON only on explicit `domain=` parameter.

**Decision**: Default response = `summary` field from artifact (already generated by
`format_summary()`). Domain-specific response = filtered JSON dict for that domain only.

### A5 — Atomic Writes (all agents flagged, already implemented)
`persist_artifact()` already uses `.tmp` + `tmp.replace()` (os.rename) — this is correct.
Dashboard and MCP tool can safely read while writer is working.

### A6 — Freshness Thresholds Per Domain (Codex + Qwen3)
The artifact is a point-in-time snapshot. Consumers should check `generated_at` against
domain-specific thresholds before acting on data.

| Domain | Staleness threshold | Notes |
|--------|---------------------|-------|
| services | 60s | systemd state changes fast |
| errors | 300s (5min) | coordinator logs rotate slowly |
| git | 900s (15min) | matches timer interval |
| validation | 1800s (30min) | focused-CI runs less often |
| data_topology | 300s (5min) | Qdrant/PG sizes change slowly |
| agent | 60s | RESUME.json changes frequently |
| performance | 900s (15min) | aq-report is slow to generate |
| attention | 120s (2min) | human gate items are urgent |

### A7 — Partial Failure Semantics (Codex)
If a source is unavailable, the artifact MUST still be generated with:
- `"<domain>": { "_source_status": "unavailable", "_error": "<msg>" }`
- `"partial": true` at the top level
Never fail the whole artifact because one source is down.

### A8 — AppArmor Coverage (Gemini, Codex, Qwen3)
Two AppArmor additions required before deployment:
1. `hybrid-coordinator` profile: add `r` for `latest-system-state.json`
2. `command-center-dashboard` profile: add `r` for same path
The `ai-system-state.service` itself runs unconfined (no profile exists yet; track as P3).

### A9 — Token Efficiency (user direction, all agents)
The MCP tool is the token-saving mechanism. All agents must:
1. Call `context_system_state(domain="<specific>")` not bare `(domain="all")`
2. The summary field is injected into `aq-session-start` context — never the full JSON
3. Dashboard reads from the artifact directly (no extra API calls)

**Registration**: `router.py` tool registry
**Implementation requirements**:
- `async def context_system_state(domain: str = "summary", max_age_s: int = 900) -> dict`
- File I/O MUST use `asyncio.to_thread()`
- Default `domain="summary"`: return `{"summary": <format_summary text>, "generated_at": ..., "freshness_seconds": ..., "partial": ...}`

- If artifact missing or older than `max_age_s`: return last available + `"_stale": true`; do NOT block to generate
- Log tool call to AIDB `system-state` collection (best-effort, non-blocking)
- Return a `summary` field of ≤200 tokens in all cases

- `context_system_state(domain="services")` returns all services
- `context_system_state(domain="summary")` returns ≤200-token text
- Handles missing artifact gracefully
- `asyncio.to_thread` verified (no sync I/O in async handler)

### QA Coverage (Service Coverage Contract)
Per WORKFLOW-CANON.md, both an `aq-qa` check AND a dashboard panel are required before
Phase 115 is "done". Add to focused CI:
- `system-state-artifact-exists`: checks artifact is ≤ 920s old after timer fires
- `context-system-state-tool`: tests the MCP tool returns valid domain data

### AIDB Upsert
Each artifact generation (in aq-system-state) should optionally upsert a condensed summary
to Qdrant collection `system-state`. Implement as best-effort (non-blocking, skip on failure).
This enables semantic queries: "what errors occurred last week?"

### No New Dependencies
Do not add Pydantic, filelock, or other new Python packages to aq-system-state.
Use stdlib `fcntl` for locking. Use plain dicts for the artifact schema.
(Qwen3 suggested Pydantic — rejected: adds package overhead to a CLI tool.)
