# Requirements — HTML Learning / Educational Module Documents

Orchestrator: claude-opus-4-8, 2026-07-06. Scope: define what the recently-created HTML
educational docs (`assets/education/*.html`, `assets/modules/*.html`, generated by
`scripts/ai/generate-module-dashboard.py`) MUST contain. The current set is reference/CS-
theory only and omits most of the harness's stated needs and focus.

## Current state (what exists)
- ~18 module pages + 3 education pages, styled HUD theme, Mermaid diagrams.
- Per-module data model has exactly 3 fields: `function`, `implementation`, `grounding`
  (CS theory) + external academic links.
- Gaps (verified): NO page covers the stated strategic focus — A2A safeguards, the
  FUTURE-AGENTIC-SURFACES targets, the multi-agent ops window, maturity/trust lifecycle;
  NO learning pedagogy (objectives/exercises/pitfalls); NO harness-governance framing;
  external academic links but NO links to our own SSOT; implementation prose is
  unverified against code (accuracy/anti-gaming risk).

## R1 — Required per-module content dimensions (expand the 3-field model)
Every module page MUST render these sections (missing ones are the gap):
1. **Learning objectives** — 3-5 "after this you can…" statements. *(missing)*
2. **Prerequisites** — prior modules / concepts needed. *(missing)*
3. **Function** — what it does. *(present)*
4. **Implementation** — how, with real file:line anchors. *(present, must be verified — see R4)*
5. **CS/theory grounding** — the underlying principle. *(present)*
6. **Harness constraints & principles** — which NON-NEGOTIABLES govern it: NixOS
   declarative-only (Rules 13/14), ports SSOT (`options.nix`), local-first, anti-gaming,
   dashboard parity, security gate. *(missing — the user's core focus)*
7. **Security & trust posture** — sandbox confinement, secret handling, trust tier, failure
   modes, zero-trust boundaries. *(missing — heavy harness focus)*
8. **How to use & verify** — the exact invocation, the `aq-qa` check that validates it, how
   it surfaces on the dashboard (parity). *(missing — turns reference into operational skill)*
9. **Worked example / hands-on** — one concrete run with expected output. *(missing)*
10. **Common pitfalls** — from `.agent/PROMOTED-BUG-PATTERNS.md` relevant to this module. *(missing)*
11. **Maturity & provenance** — lifecycle state (backlog→ready-for-prd→promoted) + trust/
    license provenance. *(missing — ties to Software Factory readiness gaps)*
12. **Canonical SSOT links** — link to our governing docs (CLAUDE.md, AGENTS.md,
    WORKFLOW-CANON, options.nix, the module's real source) IN ADDITION to academic refs. *(missing)*

## R4 — Accuracy / anti-gaming constraints (HARD)

  approximate/hallucinated prose. Ports/URLs MUST match `nix/modules/core/options.nix`, never
  hardcoded guesses (current pages contain unverified port claims, e.g. affective-engine ":8003").
- A generator validation step MUST fail if a referenced file/symbol does not exist.
- Content is generated + regenerated from a data model; do NOT hand-edit the HTML (matches the
  generated-artifact discipline — treat like `harness-prompt-extensions`).
