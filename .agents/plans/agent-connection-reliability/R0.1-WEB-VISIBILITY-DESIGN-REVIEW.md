## Review history and accepted decisions

The first review requested six binding revisions: bounded concurrent pipe draining, explicit timestamp
authority, failure-class cache precedence, removal of circular web-health self-attestation, corrected
parser/adapter authority, and proof that the deployed dashboard service context can execute the
adapter. A second review found that the facts contract contained no upstream timestamp and that
cancelled pipe readers were not explicitly awaited. The final candidate closes all eight findings.

- `task_registry.py` remains the sole compatibility JSONL parser; the TUI is only the sanitized web
  adapter, and the backend is forbidden from accessing or parsing the registry.
- Both child pipes are streamed concurrently under 8 KiB/2 KiB bounds. All failure paths converge on
  terminate, bounded wait, kill, reap, and bounded awaiting of both reader tasks.
- Backend `accepted_at_monotonic` exclusively controls 15-second freshness and the 60-second stale
  ceiling. `generated_at` is backend UTC acceptance display time; it is not an upstream fact or cache
  authority.
- Integrity, privacy, schema, JSON, and overflow failures invalidate/bypass cache. Only operational
  failures can use bounded stale evidence.
- Phase-0, deployed asset/revision evidence, service-context execution, and independent acceptance
  establish web coverage; the payload cannot self-attest.
- Exact service user, interpreter, imports, AppArmor, working directory, and mount context are an
  acceptance gate. Any required Nix, wrapper, service, or packaging change stops the nine-file slice.

This is design acceptance only. It authorizes no dashboard/runtime edit, deployment, registry read or
write, route activation, traffic cutover, staging, or commit. The seven-file R0.1 foundation must land
first under a separate authorization; this nine-file visibility slice then requires its own fresh
hash-bound authorization and live acceptance.

VERDICT: PASS
