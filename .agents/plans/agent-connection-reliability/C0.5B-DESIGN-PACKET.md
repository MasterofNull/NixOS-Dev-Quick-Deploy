## Objective

Extend the pure Agent Ops projection with injected C0.5A review-receipt and learning-candidate health
facts. The projection exposes baseline, roster, verdict, revision, finding, promotion, and freshness
state without reading or mutating any collaboration round, registry, issue, memory, prompt, service,
or provider.

C0.5B does not adjudicate a review round, transition a learning candidate, create lifecycle authority,
or claim a live broker. C0.5A remains the semantic authority for review and feedback decisions.

## Exact implementation inventory

Only these three files may change in a future separately authorized implementation:

1. `config/schemas/agent-ops-projection.schema.json`
2. `scripts/ai/lib/agent_ops_projection.py`
3. `scripts/testing/test-agent-ops-projection.py`

Any fourth implementation file is a stop condition.

C0.5B may not modify fixtures, TaskRegistry, `aq-delegation-registry`, collaboration-round models,

## M2A predecessor protection

The future implementation authorization must bind the frozen committed predecessor hash of
`scripts/testing/test-agent-ops-projection.py` recorded above.

Tests M2A.33–M2A.41 are frozen regression evidence and must remain semantically unchanged. They prove
that `show_m2a`:

- creates no writer lock or directory;
- never enters a writer transaction;
- opens no write-capable descriptor;
- rejects symlink and non-regular registries;
- reads one opened inode across atomic replacement;
- preserves malformed and record-bound failures;
- leaves the registry tree unchanged;
- returns whole snapshots under concurrent atomic replacement.

C0.5B projection logic must never call `TaskRegistry.show_m2a`, import `task_registry.py`, or read
registry state. Its facts are injected by its caller.

## Injected fact contract

`project_agent_ops()` accepts an optional `review_feedback_facts` mapping. The caller supplies
already-sanitized, already-adjudicated facts. The module validates their bounded shape and internal
consistency but does not recompute C0.5A adjudication.

The injected object contains:

```json
{
  "contract_version": "aq.review-feedback-facts.v1",
  "receipt_schema_version": "aq.review-round-receipt.v1",
  "candidate_schema_version": "aq.learning-candidate.v1",
  "subject_hash": "<sha256>",
  "pass_id": "<bounded id>",
  "baseline_hash": "<sha256>",
  "baseline_state": "aligned",
  "terminal_decision": "accepted",
  "revision": 1,
  "superseded": false,
  "binding_required": 2,
  "binding_received": 2,
  "roster_complete": true,
  "required_lanes": 2,
  "received_lanes": 2,
  "parked_lanes": 0,
  "unavailable_lanes": 0,
  "abstained_lanes": 0,
  "oldest_wait_s": 0,
  "open_findings": 0,
  "critical_undisposed": 0,
  "disposed_findings": 0,
  "promotion_state": "not_assessed",
  "promotion_pending": 0,
  "feedback_freshness": "fresh",
  "feedback_freshness_age_s": 0,
  "lanes": [],
  "reason_codes": []
}
```

Lane summaries are bounded to 64 entries and expose only:

- sanitized lane key;
- role;
- model tier: `flagship`, `economical`, `local`, or `embedded`;
- eligibility: `advisory`, `binding_flagship`, or `recused`;
- state;
- typed verdict or null.

They expose no prompt, output, path, principal credential, raw error, or provider payload.

## Default semantics

When `review_feedback_facts` is absent, the projection must return:

- `health = unavailable`;
- `assessment = not_assessed`;
- `baseline_state = not_assessed`;
- `terminal_decision = not_assessed`;
- `finding_state = not_assessed`;
- `promotion_state = not_assessed`;
- `freshness = unavailable`;
- every count and age as `null`, never zero;
- `reason_codes = ["review_feedback_not_assessed"]`.

Absence is not success, abstention, an empty roster, or zero findings.

`failed`, `timed_out`, `parked`, and `unavailable` lanes are terminally accounted but never counted
as received binding review. Explicit `abstain` remains distinct and never satisfies quorum.

Health derivation is deterministic:

- `unavailable`: facts absent.
- `blocked`: malformed or inconsistent facts, baseline mismatch, rejected decision, critical
  undisposed finding, or invalid supersession.
- `degraded`: incomplete or revision-required decision, parked or unavailable required lane,
  incomplete roster, or stale feedback.
- `healthy`: aligned baseline, complete roster, satisfied binding quorum, accepted decision, no
  blocking finding, valid revision, and fresh disposition evidence.

## Low-cardinality metrics

Add nullable numeric metrics:

- `review_required_lanes_count`
- `review_received_lanes_count`
- `review_parked_lanes_count`
- `review_unavailable_lanes_count`
- `review_abstained_lanes_count`
- `review_oldest_wait_seconds`
- `review_revision_count`
- `review_open_findings_count`
- `review_critical_undisposed_count`
- `feedback_promotion_pending_count`
- `feedback_freshness_age_seconds`

No subject hash, pass ID, lane key, model name, prompt, path, finding text, or raw reason may become a
metric name or label.

Existing Agent Ops metrics remain unchanged. Tests that currently iterate `METRICS` must distinguish
legacy fixture-backed metrics from nullable C0.5B metrics because the reserved inventory does not
permit changing the existing golden fixture.

## Purity boundary

The new C0.5B path must perform no:

- filesystem read or write;
- registry or round lookup;
- process creation or inspection;
- subprocess execution;
- socket or network access;
- environment lookup;
- system or monotonic clock call;
- randomness;
- provider or model call;
- telemetry write;
- lifecycle transition.

Time and age values are injected. Inputs are copied or treated immutably. The projection returns data
only.

Existing M0/M2A tests may continue using temporary filesystem and subprocess fixtures; C0.5B-specific
tests may not.

## Required test vectors

1. No facts produces exact `unavailable/not_assessed` output with nullable counts.
2. Complete aligned roster with independent binding acceptance produces `healthy`.
3. Missing required lane produces `incomplete/degraded`.
4. Parked lane is terminally accounted but does not satisfy quorum.
5. Unavailable lane differs from explicit abstention.
6. Baseline mismatch produces `blocked`.
7. Subject, policy, roster, or criteria drift cannot appear healthy.
8. Self-review or material-rewriter receipt cannot appear binding.
9. Accepted decision with insufficient binding quorum fails closed.
10. Revision-required receipt exposes revision and degraded health.
11. Superseded receipt cannot remain current or healthy.
12. Critical undisposed advisory finding produces blocked health.
13. Typed finding disposition removes the undisposed count without erasing history.
14. Promotion states and freshness project deterministically.
15. Missing promotion facts remain `not_assessed`, not promoted.
16. Unknown nested fields, invalid enums, malformed hashes, negative counts, oversized arrays, and
    max-plus-one strings fail closed.
17. Inconsistent totals fail closed, including received greater than required and roster complete
    with missing lanes.
18. Permuted lane order produces identical canonical output and digest.
19. Sensitive keys and raw prose are rejected.
20. Input mappings remain byte-equivalent after projection.
21. Patching filesystem, process, socket, environment, clock, and randomness entry points to raise
    does not affect C0.5B projection.
22. Draft 2020-12 schema validates positive vectors and rejects unknown top-level and nested fields.
23. Every metric is numeric or null and uses a fixed bounded name.
24. Existing M0, M1, M2A, dispatch-contract, and C0.5A tests remain green.
25. M2A.33–M2A.41 remain present and passing without projection calling `show_m2a`.
26. `contract_health()` exposes review/feedback health without treating absent facts as healthy.
27. Repeated projection of identical injected facts produces identical canonical digest.
