## 1. Objective and boundary

C0.6 defines a bounded compatibility repair for the legacy `delegate-to-local --mode direct` path:
an explicit timeout becomes an immutable end-to-end lifecycle deadline, queue and inference phases
receive separate bounded budgets, every ordinary exit converges to visible terminal evidence, and
local slot ownership is always released by the owning process.

The repair must align the legacy entrypoint with the accepted local-inference contract and the future
Agent Connection Reliability local adapter. It must not create another lifecycle store, activate the
broker, cut over traffic, change model weights, deploy services, or claim resilience against caller
death, `SIGKILL`, daemon restart, or namespace teardown. Those guarantees remain owned by the future
host-side broker.

This packet authorizes nothing. Independent flagship design review, a fresh exact-inventory
authorization, implementation evidence, exact-hash acceptance, and an explicit activation decision
are required before any live behavior changes.

## 3. Root cause chain

### 3.1 Undeclared wrapper override defeats task profiles

`delegate-to-local` unconditionally exports `DIRECT_MAX_TOKENS=4096`. That variable is absent from
`config/env-contract.yaml`, yet `TaskConfig` gives it precedence over the correctly classified
`research` profile's smaller output hint. The incident progress receipt confirms the resulting
4096-token budget.

The existing L2A policy already defines bounded review work as 250 output tokens, a 60-second
deadline, and an 8-second queue wait. The live legacy delegation path did not consume that contract.

### 3.2 Explicit 180 seconds becomes 4,156 seconds

Direct dispatch calls `_scale_timeout(explicit_timeout, max_tokens)`. At the configured conservative
floor of one token per second, 4,096 tokens plus 60 seconds of headroom changes 180 seconds into:

```text
max(180, ceil(4096 / 1.0) + 60) = 4156 seconds
```

The explicit timeout is therefore treated as a lower bound. The wrapper still prints 180 seconds,
making its operator-visible claim inconsistent with execution.

### 3.3 One expanded duration is reused as multiple independent waits

The expanded 4,156 seconds is independently supplied to:

1. banded slot-queue acquisition;
2. `urllib.request.urlopen()` for the inference request; and
3. the raw response socket's blocking read timeout.

These are phase-local waits, not one lifecycle deadline. Queue time, connection/header/prefill time,
and generation silence can each consume the duration. A stream that periodically emits chunks can
extend total wall time indefinitely because no absolute deadline is checked per chunk. `--wait` adds
no outer bound.

### 3.4 Pre-header work is operationally dark

`DirectRunner` opens the output file only after `urlopen()` returns response headers. A task that owns
the scheduler slot but stalls in connection, server queueing, or prefill has no output artifact and
retains its last queue progress state. Operators cannot distinguish queued work from prefill/header
silence.

### 3.5 Slot cleanup and terminal convergence are not structural

Slot release is invoked on selected normal and exception branches rather than one ownership-scoped
`finally`. Cancellation or an unhandled boundary can retain a queued/running marker. Dead-PID garbage
collection is derived by the scheduler reader but is not necessarily durably saved by observation.

`dispatch_task()` writes registry and PENDING terminal state only after `runner.run()` returns. Its
caller has a `finally` for trace closure, not task terminal convergence. Unexpected runner errors,
signals, or process loss can leave a `running` registry row, no terminal audit event, no useful output,
and stale scheduler ownership.

## 4. Required deadline semantics

### 4.1 One immutable lifecycle deadline and restart projection

The trusted ingress records both a process-local monotonic lease and the portable inputs needed to
reconstruct a conservative lease after process or broker restart:

```text
persisted: admitted_epoch_ms, timeout_duration_ms
in process: deadline_monotonic = admitted_monotonic + timeout_duration_ms
remaining_ms = deadline_monotonic - monotonic_now
```

No downstream phase, token estimate, environment variable, retry, progress event, or model output may
extend the accepted duration or `deadline_monotonic`. The caller-visible timeout and effective
lifecycle duration must agree. A monotonic timestamp is meaningful only within the process/boot that
sampled it and therefore must never be serialized as portable authority.

The compatibility path persists `admitted_epoch_ms` and `timeout_duration_ms` in its existing bounded
task evidence before queue acquisition; this is evidence, not a second lifecycle store. When the
future broker owns the existing lifecycle record, it reads those two values after restart, computes
`restart_remaining_ms = max(0, admitted_epoch_ms + timeout_duration_ms - wall_epoch_now_ms)`, and
establishes a new process-local lease as `monotonic_now + restart_remaining_ms`. It must never restore
the original full duration. If the wall clock moved backwards, the persisted values are absent or
invalid, the boot/lease identity is ambiguous, or elapsed time cannot be bounded conservatively, the
broker closes the task as `executor_lost` with `phase_restart` and `restart_deadline_uncertain`; it does
not resume or respawn provider work. Persisting these fields in the broker's one lifecycle record may
require a separately reviewed amendment to that store's accepted schema; it does not authorize a new
store here.

Throughput calibration remains evidence for admission and estimation only. If output work cannot fit
the deadline, resolution must select the policy/profile output cap, accept an explicit compatible
budget, or fail before provider work with canonical reason `admission_denied` and evidence code
`budget_deadline_incompatible`. It must not silently enlarge the deadline.

### 4.2 Reserved cleanup budget and phase budgets

The existing L2A policy remains the source for task/caller budgets. No new timeout policy is created.
After policy resolution, the adapter derives a non-configurable safety reserve:

```text
cleanup_reserve_ms = min(5000, max(1000, floor(timeout_duration_ms * 0.05)))
work_deadline = deadline_monotonic - cleanup_reserve_ms
```

Admission fails with canonical reason `admission_denied` and evidence code
`budget_deadline_incompatible` when `work_deadline <= admitted_monotonic` or the accepted output budget
cannot fit before `work_deadline` under the declared conservative throughput floor. Queue and provider
phases are bounded by `work_deadline`, not the full lifecycle
deadline, so they cannot consume cleanup authority. Cleanup grace means only the reserved tail between
`work_deadline` and `deadline_monotonic`; it is not extra time after the caller-visible deadline, may
not be renewed, and is used only for owned slot release, bounded artifact close/fsync, and terminal
evidence convergence. Cleanup operations are idempotent and individually bounded by the remaining
grace. At `deadline_monotonic`, the wrapper exits nonzero even if convergence is incomplete, records
`integrity_failed`/`phase_cleanup` where a bounded write remains possible, and leaves durable repair to
the named reconciler or future broker. It must not block beyond the lifecycle deadline.

The legacy adapter projects the accepted fields as follows:

| Phase | Bound | Required outcome |
|---|---|---|
| Admission/context | time before `work_deadline` | Fail before queue/provider work when the request cannot fit or is unauthorized. |
| Queue | `min(policy.queue_wait_ms, work_deadline - now)` | On exhaustion, remove the exact queued owner and publish canonical `timeout` plus queue evidence codes. |
| Connect/header/prefill | declared phase cap and `work_deadline - now` | Publish visible `starting`/`prefill` progress before the blocking open. |
| Generation silence | declared silence cap and `work_deadline - now` | Preserve bounded partial output and publish typed incomplete/timeout evidence. |
| Streaming | check `work_deadline` for every chunk | Progress cannot renew total authority. |
| Cleanup | reserved tail ending at `deadline_monotonic` | Release owned resources and converge bounded terminal evidence. |

Queue, prefill, and generation metrics are observations, not lease-renewal authority.

### 4.3 Deterministic policy and budget resolution

- Remove the wrapper's `DIRECT_MAX_TOKENS` export and remove that undeclared alias from TaskConfig.
- Do not add `DIRECT_MAX_TOKENS` to the environment SSOT.
- Add explicit wrapper forwarding for `--max-tokens`.
- Resolve output budget from explicit CLI value, accepted local policy/profile, then only a declared
  canonical override where its consumer contract allows it.
- Preserve task class and caller-tier provenance in bounded evidence. A review request must not fall
  back to a mode-global 4,096-token budget.

Resolution is deterministic and fail-closed:

1. Normalize the modal aliases `research`, `analysis`, `analysis_only`, and `research_only` to the L2A
   `review` task class only for read-only work, the `coding_logic` lane, allowed tools, and a domain not
   listed in `blocked_domains`. `planning` and `prd` receive that mapping only when the classified
   domain is not architecture or policy; otherwise local admission is denied. Modal tuning does not
   grant a broader policy class.
2. Authenticate caller tier; never trust a prompt-asserted tier. Intersect caller-tier, task-class,
   selected-profile snapshot, and explicit request budgets componentwise. For the current review
   mapping this yields no more than 250 output tokens, 60,000 ms lifecycle, and 8,000 ms queue wait,
   even when the caller tier permits more.
3. An omitted value receives the componentwise policy cap. An explicit value at or below the cap is
   accepted unchanged. An explicit output or deadline above the cap is rejected as
   `admission_denied`/`requested_budget_above_policy`; it is never silently enlarged or truncated.
4. A longer task is accepted only when an authenticated caller tier, eligible named task class, and
   immutable profile snapshot all explicitly authorize that value. Selecting a higher tier in CLI
   text, prompt content, environment, or an alias is insufficient. No such authorization means reject,
   not fallback or automatic remote rerouting.

Timeout precedence is also deterministic. An already accepted external cancellation wins over a
timeout observed later; otherwise the first atomic in-process terminal claim wins. Queue-policy expiry
before `work_deadline` is a timeout with queue phase evidence; provider phase expiry before
`work_deadline` is a timeout or incomplete-output result as defined below; `work_deadline` always starts
cleanup; and `deadline_monotonic` always ends the process. Wall-clock timestamps are evidence only
after admission and never override the active monotonic lease.

## 5. Execution and convergence design

### 5.1 Direct runner

Before any blocking provider open, DirectRunner creates the declared output/progress surfaces and
publishes a bounded phase state. It passes only the remaining deadline to blocking operations and
checks the absolute deadline during streaming. Timeout artifacts distinguish queue, prefill/header,
generation silence, and lifecycle exhaustion without including prompts, raw provider errors, URLs,
paths, PIDs, or model output in metric labels.

The output artifact is created in a trusted task directory using symlink-safe exclusive creation with
mode `0600`; inherited umask may make it stricter but never broader. Writes stop at the accepted C0
`budget.output_bytes` cap (and the resolved output-token cap), with bounded truncation metadata stored
separately from content. A partial artifact is always marked incomplete and unaccepted and is subject
to the existing evidence-retention/redaction policy. Access is limited to the owning account, the
broker's authorized artifact collector, and an explicitly authorized reviewer; the lifecycle
registry itself receives metadata only. Prompts, output fragments, hashes derived
from prompt/output content, and raw provider errors are forbidden from metrics, dashboard payloads,
audit labels, and progress summaries. Partial content never appears in Agent Ops cards. Partial output
remains evidence but cannot be accepted as success; terminal classification uses the C0 reason
`output_incomplete` with bounded phase/code evidence.

### 5.2 Slot ownership

`slot_queue` exposes one idempotent cleanup operation bound to the exact process/run identity. It may
remove only that owner's queued or running record. DirectRunner invokes it from one outer `finally`
after ownership may have been created. Timeout and ordinary exceptions must leave neither a queued
nor running marker.

Dead-owner reconciliation must durably persist the derived cleanup when it is explicitly performed.
Read-only monitoring remains non-mutating; it reports stale ownership until a named reconciler or the
future broker repairs it.

### 5.3 In-process terminal convergence and deferred external fencing

The dispatch boundary converts ordinary timeout and exception paths into:

1. one bounded output/progress terminal-artifact attempt;
2. one idempotent registry terminal-transition attempt;
3. one idempotent PENDING-completion attempt;
4. one idempotent terminal-audit attempt; and
5. a nonzero wrapper exit for failure.

The exactly-once claim is deliberately narrow: one live process serializes its ordinary timeout/error
paths through one terminal arbiter and invokes each terminal sink at most once. This does not prove
cross-process exactly-once delivery or durable cancellation. External cancellation versus late
provider completion requires the broker's CAS revision, lease owner, and fencing epoch, or a separately
authorized amendment to the existing lifecycle authority. Until that exists, the compatibility path
must not advertise externally fenced cancellation; it may only observe an already accepted
cancellation and refuse a later in-process success. `SIGKILL`, caller teardown, PID namespace loss,
restart, and ambiguous sink acknowledgement converge through broker reconciliation, not this slice.

### 5.4 C0 reason and evidence-code mapping

C0's closed `reason` enum is the only terminal taxonomy. Phase and detail are bounded entries in the
existing `evidence_codes` array; they are not new reasons or free-form labels.

| Condition | C0 reason | Required evidence codes |
|---|---|---|
| Ineligible/oversized admission | `admission_denied` | `phase_admission`, `requested_budget_above_policy` or `budget_deadline_incompatible` |
| Policy/domain/tool rejection | `policy_blocked` | `phase_admission`, `local_policy_intersection_empty` |
| Queue wait exhausted | `timeout` | `phase_queue`, `queue_wait_exhausted` |
| Connect/header/prefill exhausted | `timeout` | `phase_prefill`, `provider_headers_deadline` |
| Generation silence with no output | `timeout` | `phase_generation`, `generation_silence` |
| Deadline after any output byte | `output_incomplete` | `phase_generation`, `partial_output_deadline` |
| Accepted external cancellation | `cancelled` | `phase_cancellation`, `owned_cancel_accepted` |
| Cleanup/convergence grace exhausted | `integrity_failed` | `phase_cleanup`, `terminal_convergence_incomplete` |
| Restart/ownership uncertainty | `executor_lost` | `phase_restart`, `restart_deadline_uncertain` or `owner_reconciliation_required` |

Metric and dashboard vocabularies project only these canonical reasons and bounded phases. A legacy
phase-specific reason presented at a migration boundary must be deterministically normalized to the
listed canonical reason plus evidence codes or rejected; it is never emitted or stored as a parallel
top-level taxonomy.

## 7. Thirteen required test families

1. **Immutable explicit deadline:** `--timeout 180` remains 180 seconds even with a 4,096-token
   estimate or inherited legacy environment value.
2. **Policy/profile budget:** a review/research prompt resolves the declared bounded profile/policy
   output budget and is never forced to 4,096 by the wrapper.
3. **Aggregate lifecycle:** a simulated 60-second queue wait plus header stall terminates at the one
   absolute deadline, not queue deadline plus request timeout or 4,156 seconds.
4. **Queue independence:** queue policy exhaustion produces canonical `timeout` with
   `phase_queue`/`queue_wait_exhausted` evidence and leaves no queued or running owner.
5. **Claimed prefill timeout:** a task that claims the slot and receives no headers releases ownership
   exactly once and writes typed output/progress evidence.
6. **Streaming deadline:** periodic SSE chunks cannot extend the lifecycle deadline; partial output is
   preserved and classified incomplete.
7. **Exception convergence:** an unexpected ordinary runner exception invokes each registry, PENDING,
   audit, and artifact terminal sink at most once and emits bounded convergence evidence.
8. **Pre-header visibility:** output/progress surfaces exist before `urlopen`, and projection shows
   `starting`/`prefill` rather than stale queued/no artifact.
9. **Terminal race:** timeout/cancellation versus late provider completion yields one terminal result;
   late success cannot overwrite it. Durable cross-process fencing remains a broker acceptance test.
10. **Durable stale-owner repair:** named reconciliation persists dead queued/running cleanup without
    relying on a later admission; read-only monitoring itself remains non-mutating.
11. **Blocking wrapper behavior:** `--wait` exits nonzero no later than the immutable lifecycle
    deadline; queue/provider work stops at `work_deadline` and only the reserved internal cleanup tail
    is consumed before exit.
12. **Long-task explicitness:** long tasks remain supported only with an explicit compatible deadline;
    no regression test expects automatic timeout enlargement.
13. **Regression and parity:** local inference budget, artifact, slot queue, L2A/L2B delegate-to-local
    versus aq-chat parity, Agent Ops, local reliability, and Phase-0 suites remain green.

Tests use fake clocks, fake slot/provider adapters, bounded temporary registries, and deterministic SSE
fixtures. Unit tests must not call the live model. One harmless live canary is reserved for the
activation gate.

### 7.1 Required boundary and adversarial vectors

Each applicable family above must include deterministic vectors for:

- timeout durations immediately below, at, and immediately above the cleanup-reserve/minimum-work
  boundary; zero, negative, non-integer, overflow, and maximum schema values fail closed;
- a wall-clock rollback, wall-clock forward jump, reboot/monotonic reset, missing persisted admission
  field, corrupt duration, and restart after expiry; no vector restores the original full duration;
- queue completion one tick before `work_deadline`, exactly at it, and one tick after it; connect,
  header, prefill, SSE silence, and chunk arrival at the same three boundaries;
- cleanup success on the final permitted tick, cleanup sink timeout, fsync failure, repeated cleanup,
  and an operation attempting to block beyond grace;
- explicit budgets below/at/above each task-class, caller-tier, and profile cap; unauthenticated or
  prompt-asserted flagship tier; missing profile snapshot; blocked domain; disallowed tool; and empty
  policy intersection;
- every research alias mapped to eligible read-only review work, plus `planning`/`prd` architecture or
  policy requests that must be denied rather than implicitly rerouted;
- queue timeout, prefill timeout, zero-byte generation silence, partial-output timeout, accepted
  cancellation, cleanup failure, and uncertain restart producing only the exact C0 reason and bounded
  evidence-code pairs in section 5.4; unknown reason/phase/code values fail closed;
- timeout racing ordinary exception, late success, and an already accepted cancellation in every
  ordering; tests assert only in-process at-most-once sink invocation and do not claim external fencing;
- output path pre-existence, symlink substitution, permissive umask, short write, disk-full/fsync error,
  exact byte cap, one-byte overflow, multibyte UTF-8 boundary, and partial content containing privacy
  canaries; mode remains `0600` or stricter and no content reaches telemetry/dashboard/audit;
- dead PID reuse, mismatched run identity, repeated reconciliation, and read-only projection during
  stale ownership; only the named owner/reconciler mutates state;
- wrapper interruption before artifact creation, after slot claim, during pre-header silence, during
  streaming, and during cleanup; every ordinary catchable path exits bounded and leaves typed evidence;
- Track A tests passing while dashboard/Phase-0 fields are absent; completion and activation must remain
  false until the separately accepted C0.6-T service-coverage evidence is present.

### Rollback

Rollback is a versioned routing/configuration decision, not destructive Git or filesystem mutation.
If timeout rate, incomplete output, scheduler leakage, dashboard gaps, or parity regressions exceed
the accepted threshold:

1. stop new canary admissions;
2. allow already admitted work to converge or cancel it through owned authority;
3. restore the prior declared legacy-direct behavior using the reviewed rollback target;
4. retain all receipts and evidence;
5. reconcile stale ownership through the named operation; and
6. record the failure as a regression fixture before reconsidering activation.

Rollback may not re-enable undeclared environment overrides, hide telemetry, delete evidence, or
silently redirect the intended lane.
