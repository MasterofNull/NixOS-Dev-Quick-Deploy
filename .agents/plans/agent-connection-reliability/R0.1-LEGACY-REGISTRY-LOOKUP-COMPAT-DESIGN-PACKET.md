# R0.1 Legacy Registry Lookup Compatibility — Design Packet

Status: **PREPARED_ONLY / DESIGN_ONLY — NO IMPLEMENTATION OR DATA MUTATION AUTHORITY**
Date: 2026-07-16
Parent: `.agent/PROJECT-AGENT-CONNECTION-RELIABILITY-PRD.md`
Risk tier: **R2 / medium** — read-only compatibility path over security-sensitive lifecycle evidence

## 1. Objective and boundary

Repair `aq-delegation-registry show <task-id>` so one bounded, valid oversized legacy JSONL row does
not deny lookup of an unrelated bounded record. Preserve the M2A writer's 4 KiB record invariant,
regular-file and symlink protections, atomic-inode snapshot behavior, read-only operation, duplicate-ID
detection, privacy, and fail-closed handling of ambiguity.

This design does not authorize implementation. It must not rewrite, compact, migrate, quarantine,
delete, normalize, or otherwise mutate `.agents/delegation/registry.jsonl`; weaken mutation-path
validation; add a registry or index; activate M2B; or treat a legacy oversized target as a valid M2A
record. A separately reviewed data-migration slice is required before strict M2A mutations can operate
on a mixed legacy registry.

## 2. Evidence and root cause

Read-only inspection on 2026-07-16 established:

- `.agents/delegation/registry.jsonl` is mode `0600`, 861,535 bytes, and 666 lines.
- 47 valid JSON rows exceed `_M2A_MAX_RECORD_BYTES = 4096`; the largest is 19,496 bytes.
- The first oversized row is line 45, a terminal legacy record from 2026-05-18.
- No observed row exceeds 65,536 bytes.
- `aq-delegation-registry show claude-20260716-150907-o2le6i` returns
  `{"ok": false, "error": "registry_record_too_large"}` even though that unrelated requested row is
  bounded.

The cause is structural. `TaskRegistry.show_m2a()` calls `_m2a_read_records()`, which reads the whole
opened inode, splits every line, and raises on the first line over 4 KiB before returning any records.
The caller cannot search until the complete strict parse succeeds. The 4 KiB invariant is correct for
new M2A records and mutation, but applying it globally to historical rows makes a single legacy row a
read-availability kill switch.

Existing M2A.33–M2A.41 intentionally pin `show_m2a()` and its strict behavior, including
`registry_record_too_large`. This slice must add a distinct `lookup_m2a_compatible()` primitive for
the CLI rather than silently weakening `show_m2a()`, `_m2a_read_records()`, or any writer caller.

## 3. Required compatibility behavior

### 3.1 Two bounded classes, one strict authority

- `_M2A_MAX_RECORD_BYTES = 4096` remains the only accepted M2A record bound.
- A new read-only `_M2A_MAX_LEGACY_LOOKUP_BYTES = 65536` is a parsing containment bound, not an
  accepted record size and not writer authority.
- A row over 4 KiB is compatibility-eligible only when it is valid JSON, is an object, has no M2A
  `record_version`, and has exactly one unambiguous bounded string identity from `id` or `task_id`.
- A row with both identity fields carrying different values, an invalid/unbounded identity, an M2A
  version, or more than 65,536 bytes fails the lookup as ambiguous/corrupt. It is never skipped.
- Oversized legacy JSON is parsed only long enough to validate its class and identity. Its description,
  prompt-like fields, paths, output, provider payloads, or other content are never returned, logged,
  hashed into task-visible identifiers, or projected to the dashboard.
- Every JSON object, including nested objects, is decoded with a duplicate-key-rejecting
  `object_pairs_hook`. A repeated key is `registry_record_duplicate_key`; last-key-wins parsing is
  forbidden because it can hide `id`, `task_id`, or `record_version`.

Identity semantics are frozen to the existing M2A task-ID grammar: a Python string of 1–128 ASCII
characters matching `^[a-zA-Z0-9][a-zA-Z0-9._:-]{0,127}$`. The query and every extracted `id` or
`task_id` pass the same pure validator. Comparison is exact and case-sensitive. There is no trimming,
Unicode normalization, case folding, percent decoding, path normalization, or lossy replacement. If
both identity fields exist they must be exactly equal after validation; otherwise the row is
ambiguous and the entire lookup fails closed.

### 3.2 Physical-line and source-snapshot contract

The scanner operates on physical bytes, not text iteration:

- A physical record is the byte payload before one `LF` (`0x0a`). One immediately preceding `CR`
  (`0x0d`) is accepted as the CRLF terminator and excluded from the record payload.
- A bare `CR`, an embedded raw control newline, or more than one `CR` before `LF` is rejected. JSON
  string newlines must use JSON escaping and remain payload bytes.
- A non-empty final record without `LF` is accepted and receives the same byte and JSON validation.
  An empty final segment after a terminal `LF` is ignored. Empty or whitespace-only physical records
  are ignored but their bytes still count toward the cumulative source bound.
- The 4,096- and 65,536-byte limits count record payload bytes after removing only the permitted line
  terminator. Terminator bytes count toward the cumulative file-read limit.
- The reader retains at most one chunk, one line up to 65,536 bytes plus the two possible terminator
  bytes, one bounded target record, fixed counters, and decoder scratch. It must reject an overlong
  line as soon as the bound is crossed rather than buffer the rest of that line.

The 50 MiB source bound is an actual cumulative-read bound, not only an initial `fstat` check. The
reader obtains an initial descriptor `fstat`, rejects an initial size above 50 MiB, counts every byte
returned by `read`, and requests no byte after `50 MiB + 1`. Observing the sentinel byte produces
`registry_file_too_large`. Short reads are not EOF and are accumulated normally. `EINTR` is retried
without advancing counters; other read errors become bounded source-unavailable codes.

Atomic pathname replacement remains safe because the descriptor stays on the opened inode. In-place
mutation of that same inode is not accepted as a snapshot: after EOF, a second `fstat` must match the
initial device, inode, size, `mtime_ns`, and `ctime_ns`. Early EOF before the initial size, growth,
truncation, or changed metadata yields `registry_source_changed_during_read`; no partial result is
returned. This detects the supported grow/truncate races. Same-size hostile overwrite that defeats
filesystem metadata is outside the read-only process's proof capability and remains prohibited by the
single atomic-replacement writer contract.

### 3.3 Lookup algorithm

Add a dedicated read-only compatibility scanner. It must:

1. open the registry once with `O_RDONLY | O_CLOEXEC | O_NOFOLLOW` where available;
2. validate the opened descriptor is a regular file and enforce both initial and cumulative 50 MiB
   bounds;
3. scan that one opened inode under section 3.2, never `Path.read_text()` and never a second path
   lookup;
4. strictly parse all rows at or below 4 KiB;
5. parse compatibility-sized legacy rows with duplicate-key rejection, retain only sanitized identity
   and count metadata, and discard the parsed object immediately;
6. scan to EOF even after finding the requested bounded row so a hidden duplicate or ambiguous
   oversized identity cannot be bypassed;
7. return exactly one bounded matching record only when no duplicate or ambiguous identity exists;
8. return `registry_target_legacy_oversized` without legacy content when the requested identity belongs
   to an oversized legacy row;
9. return `registry_task_not_found` only when every row was safely classified and no identity matched;
   and
10. leave registry bytes, inode, mode, ownership, `mtime`, `ctime`, directory entries, and lock path
    unchanged. Access time is not an invariant: an ordinary read may update `atime` under the host
    mount policy, and the implementation must not request privileged `O_NOATIME` to conceal that fact.

Malformed JSON, non-object rows, invalid UTF-8, overlong lines, identity ambiguity, and duplicate target
identities remain typed failures. Compatibility must not convert corruption into "not found."

Duplicate detection is deliberately target-only and memory bounded. The scanner never builds a set of
all registry identities. For every bounded or compatibility-sized legacy row it validates identity,
compares it to the already validated target, increments a target-match counter capped at two, and
retains at most one bounded matching record. A second target match in any ordering or size class is
`registry_duplicate_target`. Duplicate identities unrelated to the requested target are not
adjudicated by this lookup; strict mutation/migration tooling owns global uniqueness.

### 3.4 API and projection compatibility

- Keep `show_m2a(task_id) -> dict` strict and semantically unchanged for M2A callers and frozen
  M2A.33–M2A.41 coverage.
- Add `lookup_m2a_compatible(task_id)` returning the closed, versioned CLI shape
  `aq.registry-lookup-result.v1`: `schema_version`, `ok`, nullable bounded `record`, nullable
  `error_code`, and `registry_compatibility`. Successful CLI output remains `ok=true`; failure output
  is `ok=false` with one closed error code. No raw JSON, line content, task description, path, prompt,
  or parser exception prose may be emitted on a failure path.
- CLI exit codes are frozen: `0` bounded target returned, `2` CLI usage error, `3` safely classified
  target not found, `4` target is oversized legacy and therefore not returnable, `5` source
  integrity/ambiguity/corruption failure, and `6` source unavailable/access failure. Degraded but safe
  compatibility of unrelated rows does not change a successful exit `0`.
- The projection consumes a different closed input,
  `aq.registry-compatibility-facts.v1`, containing only sanitized health enums, bounded aggregate
  counts, the fixed compatibility limit, and closed reason codes. It never accepts or emits the CLI
  `record`, target ID, target outcome, line number, path, or parser text.
- The filesystem adapter/TUI reader constructs those sanitized facts. `agent_ops_projection.py`
  remains a pure injected-facts projector: it must not import `task_registry`, open a file, inspect an
  environment variable, call a clock, or invoke the compatibility scanner.
- `_m2a_read_records()`, `begin`, `attach-process`, `transition`, and `reconcile-m2a` remain strict and
  blocked by any oversized row. Read compatibility is not mutation compatibility.

## 4. Threat model and invariants

| Threat | Required defense |
|---|---|
| Oversized-row denial of unrelated reads | Bounded compatibility classification permits later bounded targets. |
| Oversized M2A record smuggled as legacy | Any `record_version` on an oversized row fails closed. |
| Hidden duplicate target before/after match | Scan the entire opened inode and reject duplicate identity. |
| `id`/`task_id` confusion | Require one field or identical values; conflicting values are ambiguous. |
| Duplicate JSON keys smuggle identity/version | Reject duplicate keys at every object depth; never use last-key-wins. |
| JSON/parser memory exhaustion | Preserve 50 MiB file bound and enforce 64 KiB legacy line bound before parsing. |
| Newline, Unicode, or invalid UTF-8 ambiguity | Frozen physical-line rules, strict UTF-8/JSON decode, ASCII identity grammar, exact comparison, no lossy replacement. |
| Symlink/FIFO/device substitution | `O_NOFOLLOW`, `fstat`, regular-file requirement, and one opened inode. |
| Atomic replacement during lookup | Complete old-or-new inode snapshot; no path reopen. |
| Same-inode growth/truncation or short read | Cumulative byte cap plus initial/final descriptor-stat agreement; changed source returns no result. |
| Legacy content disclosure | Never return or project an oversized legacy object; expose counts and typed codes only. |
| Reader becoming writer | No lock creation, directory creation, temp file, rename, truncate, append, chmod, or fsync. |
| Compatibility weakening mutations | Mutation methods continue to call the unchanged strict reader. |
| Metric cardinality/privacy leak | Fixed metric names and enums only; no task IDs, line numbers, byte sizes per row, paths, or content labels. |

The compatibility scanner is not a lifecycle authority. It observes one snapshot and cannot update
status, repair data, adjudicate task truth, or grant provider execution.

## 6. Test contract

Tests use temporary registries and synthetic privacy canaries only. Production registry content must
not be copied into fixtures.

Required deterministic families:

1. A bounded target after one and several valid oversized legacy rows succeeds.
2. A bounded target before an oversized row still scans to EOF and rejects a hidden duplicate.
3. An oversized legacy target returns only `registry_target_legacy_oversized` and sanitized health.
4. Payloads of 4,096, 4,097, 65,536, and 65,537 bytes are pinned independently under LF, CRLF, and an
   unterminated final line; terminator bytes do not alter the payload class.
5. Empty lines, whitespace-only lines, terminal LF, CRLF, bare CR, doubled CR, raw embedded newline,
   escaped JSON newline, and empty/non-empty unterminated final records have frozen outcomes.
6. Exact 50 MiB cumulative input succeeds when otherwise valid; a 50 MiB plus one-byte sentinel fails
   even when initial `fstat` is stale or mocked below the cap.
7. Short reads, repeated short reads, `EINTR`, non-EINTR read error, early EOF, same-inode grow,
   same-inode truncate, same-inode metadata change, and atomic pathname replacement have the outcomes
   specified in section 3.2.
8. Duplicate top-level and nested JSON keys, including repeated `id`, `task_id`, and `record_version`,
   fail closed for both bounded and legacy-sized rows.
9. Oversized M2A-versioned, missing-identity, conflicting-identity, non-string identity, non-object,
   malformed JSON, invalid UTF-8, and duplicate target rows fail closed.
10. Query and row identities cover empty, 128/129 characters, every allowed punctuation class,
    leading punctuation, ASCII case difference, surrounding whitespace, NFC/NFD lookalikes, non-ASCII,
    path-like text, and percent-encoded text; no normalization changes identity.
11. Target duplicates are tested bounded-before-bounded, bounded-before-legacy,
    legacy-before-bounded, legacy-before-legacy, and after a found target. A near-50-MiB registry with
    many unique unrelated IDs proves memory does not grow with roster cardinality.
12. A missing target returns not-found only after every row is safely classified.
13. Symlink, FIFO/directory, permission denial, file-over-cap, truncation during read, and atomic replace
   preserve existing typed behavior and whole-snapshot semantics.
14. Descriptor flags prove read-only access; no writer transaction, lock, directory, temp file, rename,
   content/mode/ownership/mtime/ctime mutation, or privileged `O_NOATIME` occurs. Tests permit
   mount-policy `atime` change.
15. Oversized privacy-canary content never appears in stdout, stderr, exceptions, projection, metrics,
   TUI, or Phase-0 detail.
16. Strict `show_m2a()` and mutation APIs still reject the same mixed registry with
    `registry_record_too_large`; only the CLI's explicitly named compatibility lookup bypasses an
    unrelated classified legacy row.
17. `aq.registry-lookup-result.v1` and `aq.registry-compatibility-facts.v1` reject each other's fields,
    unknown fields, target-specific projection fields, out-of-range counts, and unknown enums/reasons.
    Every CLI outcome asserts the exact exit code from section 3.4.
18. A purity test patches filesystem, environment, clock, subprocess, socket, and TaskRegistry entry
    points to raise while injected compatibility facts still project deterministically.
19. Concurrent atomic replacement yields a complete old-or-new snapshot and never combines rows.
20. `coverage_health.web_dashboard` remains `unavailable` and overall contract health cannot become
    healthy when only the seven-file TUI slice is present; a synthetic accepted web-coverage fact is
    required before completion.
21. C0.5B, M2A.33–M2A.41, dispatch, local reliability, Agent Ops schema, and Phase-0 regressions remain
    green; AST equivalence pins the existing M2A.33–M2A.41 tests unless an independently reviewed
    amendment explicitly replaces them.

## 7. Observability and dashboard contract

Agent Ops projection v2 remains the pure projection SSOT. Its adapter-provided
`aq.registry-compatibility-facts.v1` input projects a closed `registry_compatibility` object with:

- `lookup_health`: `healthy | degraded | blocked | unavailable`;
- `strict_mutation_health`: `healthy | blocked | unavailable`;
- nullable bounded counts for oversized legacy rows and ambiguous/corrupt rows;
- `max_legacy_bound_bytes` as a fixed contract value, not a per-row observation; and
- fixed reason codes such as `legacy_oversized_rows_present`, `legacy_identity_ambiguous`,
  `legacy_row_over_compatibility_bound`, and `registry_source_unavailable`.

The TUI Agentic Ops header/card must visibly distinguish:

- **lookup available / migration required** when unrelated bounded lookups work but strict mutation is
  blocked by legacy rows;
- **lookup blocked** for ambiguity, corruption, source denial, or over-cap content; and
- **healthy** only when no compatibility exception is required.

No-data is explicit `unavailable`, never zero or healthy. Metrics are low-cardinality counts only.
Task IDs, PIDs, line numbers, paths, row byte sizes, descriptions, prompts, output, raw JSON, and parser
text are forbidden. `aq-tui-dashboard --json` must carry the same closed projection shown on screen.

Phase-0 must create a temporary mixed registry, prove the unrelated lookup succeeds, prove strict
mutation remains blocked, validate the closed projection, and verify the visible TUI state. The check
must not read or modify the production registry. A blank or absent compatibility field blocks
acceptance.

Web dashboard applicability is **required but not implemented by this seven-file slice**. The existing
Agent Ops contract explicitly contains `coverage_health.web_dashboard`; it must remain `unavailable`,
and aggregate contract health must remain non-healthy, until a separately designed, source-backed web
API and visible card/badge project the same sanitized facts. The unrelated Intelligence/Agent Ops
drift card must not be overloaded and dashboard code must not duplicate registry parsing. Before any
implementation authorization or completion claim, a separate exact-inventory visibility amendment
must define that web route, no-data state, card/badge, privacy contract, and live check. This packet's
seven-file inventory may deliver only the TUI/machine projection foundation.

### Rollout

1. Obtain independent architecture/security/SRE review of this exact packet.
2. Record C0.5B predecessor commit `1e4826c0`, confirm no current writer lease on the seven files, and
   independently review a separate web-dashboard visibility amendment.
3. Freeze exact candidate hashes in a single-use authorization only after that visibility design is
   accepted; the seven-file implementation remains non-complete until web coverage lands.
4. Implement only against temporary synthetic registries.
5. Run all twenty-one families, parent regressions, schema validation, AST preservation, Phase-0, and
   the governance gate.
6. Obtain independent exact-hash flagship acceptance.
7. Activate only the read-only `show` compatibility route; do not activate M2A mutation or migrate
   data.
8. Verify the live TUI reports degraded-compatible lookup and migration-required mutation health while
   production registry bytes, inode, mode, ownership, `mtime`, and `ctime` remain unchanged; do not
   assert `atime` stability.
9. Keep `coverage_health.web_dashboard=unavailable` and activation blocked until the separately
   accepted web route/card reports nonblank sanitized compatibility health.

### Rollback

Rollback restores the previous strict show implementation by a reviewed code/config decision. It does
not edit registry data or erase evidence. The dashboard must then report lookup blocked rather than
healthy. Any compatibility failure becomes a synthetic regression vector before reconsideration.

Acceptance of the seven-file candidate requires unrelated bounded lookup availability, strict
ambiguity handling, unchanged writer behavior, zero production-content mutation, closed telemetry,
visible TUI health, Phase-0 coverage, and all predecessor regressions passing. Program completion and
activation additionally require accepted nonblank web-dashboard coverage.

## 9. Stop conditions

Stop if implementation requires registry rewrite/migration, an index or second store, parsing a legacy
row over 64 KiB, returning oversized legacy content, weakening strict mutation validation, skipping an
ambiguous or malformed row, reopening the path after descriptor validation, adding an eighth file,
overlapping an active writer lease, exposing high-cardinality/sensitive fields, or claiming M2B/broker
activation; permitting duplicate JSON keys; normalizing identity; retaining an all-ID set; treating
`atime` as immutable; importing the scanner into the pure projection; conflating CLI and projection
schemas; returning an undocumented exit code; claiming healthy `coverage_health.web_dashboard`
without the separate web amendment; or accepting an in-place changed source snapshot.
