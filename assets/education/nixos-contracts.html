Declarative vs. Imperative Architecture

Unlike traditional Linux distributions (Debian, Arch, Ubuntu) which update state imperatively at runtime (e.g. apt install, chmod 777, editing /etc/hosts directly), NixOS evaluates a declarative system description (written in Nix syntax) and constructs a read-only store.

Any code running under our AI harness (including agent tools and sub-agents) must respect the read-only contract. Standard NixOS configuration parameters reset modes, user configurations, and runtime settings upon each activation phase (nixos-rebuild switch).

# To apply changes permanently, edit declarative Nix: sudo nixos-rebuild switch --flake .#hyperd-ai-dev

SOPS-Nix & Secret Security

To prevent leaking private keys or credentials to public code models, this repository implements SOPS (Secrets Operations). Secrets are stored encrypted within secrets.yaml and decrypted at execution runtime strictly into user-isolated RAM space under /run/secrets/.

Agent Guardrail: Agents must NEVER write secrets directly into file configurations or append credentials to dynamic shell scripts. Relational ports and credentials should always be read dynamically from environment variables.
